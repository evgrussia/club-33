# Респекты, жалобы, звёзды, роли (Phase 2 + расширение Phase 3)

Документ описывает социальный слой клуба: как участники взаимодействуют через респекты, как модерация работает с жалобами, как формируется репутация (звёзды 0–4★) и как назначаются роли (ниши).

## Actors

- **Участник** — отправитель/получатель респекта, автор/предмет жалобы
- **Модератор / Super-админ** — модерация жалоб, корректировки
- **Бот** — приём команд из чата и из бота, лимиты, уведомления
- **Mini-app** — UI для респекта, жалобы, профиля
- **Cron / Scheduler** — обновление лимитов, пересчёт звёзд, деградация
- **AuditLog** — логирование всех операций

---

## 1. Процесс: Респект

### Триггеры (4 канала)

1. **Mini-app** (Phase 2): кнопка «дать респект» в профиле другого участника.
2. **Команда в чате клуба** (Phase 2): `@user +respect` или `/respect @user [причина]` в групповом чате.
3. **Команда в боте** (Phase 2): `/respect @user` в личке с ботом.
4. **Реакция-стикер в чате** (Phase 3): специальный кастомный стикер-реакция → бот ловит и засчитывает респект.

### Бизнес-правила

| Правило | Значение |
|---|---|
| Лимит респектов / месяц / отправитель | **30** |
| Лимит на одного получателя / месяц | **3** |
| Сброс лимитов | **1-го числа каждого месяца, 00:00 МСК** (cron-задача) |
| Самому себе | ❌ запрещено |
| Боту / системному юзеру | ❌ запрещено |
| Неактивному (`inactive`) | ❌ запрещено |
| Кандидату (без `active`) | ❌ запрещено (нет доступа в чат, mini-app) |
| Анонимность отправителя | ❌ получатель видит, кто дал респект |
| Отзыв респекта | ❌ нельзя отозвать после засчитывания |

### Swim-lane

```
| Отправитель        | Бот / mini-app                    | Получатель        | AuditLog              |
|--------------------|-----------------------------------|-------------------|-----------------------|
| Нажал «респект»    |                                   |                   |                       |
| или /respect @user |                                   |                   |                       |
|                    | Проверки:                         |                   |                       |
|                    | - отправитель active?             |                   |                       |
|                    | - получатель active?              |                   |                       |
|                    | - не сам себе?                    |                   |                       |
|                    | - месячный лимит 30?              |                   |                       |
|                    | - лимит на получателя 3/мес?      |                   |                       |
|                    | Если нарушено → отказ + причина   |                   |                       |
|                    | Если ок → +1 респект              |                   |                       |
|                    |   - increment в respects table    |                   |                       |
|                    |   - +1 балл репутации получателю  |                   |                       |
|                    |   - событие respect_given         |                   | log_event             |
|                    | Уведомление получателю            | «Тебе дали        |                       |
|                    |                                   | респект от X»     |                       |
|                    | Подтверждение отправителю         |                   |                       |
|                    | («Респект засчитан»)              |                   |                       |
```

### Cron-задача: ежемесячный сброс

```
| Время                          | Действие                                            |
|--------------------------------|-----------------------------------------------------|
| 1-го числа, 00:00 МСК          | Reset respects_monthly_used = 0 для всех            |
|                                | Reset respects_to_user_monthly = 0 для всех пар     |
|                                | AuditLog.log_task_start('respects_monthly_reset')   |
|                                | ... выполнение ...                                  |
|                                | AuditLog.log_task_end('...', success/error)         |
```

### Исключения

- **E1. Получатель `grace_period`**: респект разрешён (он ещё участник).
- **E2. Дубликат стикера-реакции**: если уже дал респект этому пользователю недавно, бот игнорирует повторный стикер (антиспам).
- **E3. Спам-стикеров**: если за минуту >5 стикеров-реакций от одного отправителя — модератор получает алерт.

## 2. Процесс: Жалоба

### Триггер

Участник нажимает «Пожаловаться» в mini-app (профиль другого) или вводит `/complain @user` в боте.

### Бизнес-правила

| Правило | Значение |
|---|---|
| Анонимность для получателя | ✅ получатель **никогда** не узнаёт, кто на него пожаловался |
| Видимость жалобы получателю | ❌ скрытая до момента, когда модератор её подтвердил *и* решил уведомить |
| Лимит жалоб от одного на одного | **3 / месяц** (защита от харассмента модерацией) |
| Минимум текста | **30 символов** причины |
| Состояния жалобы | `submitted` → `under_review` → `confirmed` / `rejected` |
| SLA реакции модератора | **≤48 часов** |

### Swim-lane

```
| Автор жалобы       | Бот / mini-app                | Модератор        | Получатель жалобы   |
|--------------------|-------------------------------|------------------|---------------------|
| Заполнил форму:    |                               |                  |                     |
| - на кого          |                               |                  |                     |
| - категория        |                               |                  |                     |
| - текст ≥30 симв.  |                               |                  |                     |
|                    | Проверки лимитов, валидация   |                  |                     |
|                    | Создал жалобу,                |                  |                     |
|                    | состояние submitted           |                  |                     |
|                    | Уведомление в админ-чат       |                  |                     |
|                    | (только модераторам)          |                  |                     |
|                    |                               | Открыл карточку  |                     |
|                    |                               | под_review       |                     |
|                    |                               | Возможна         |                     |
|                    |                               | переписка с      |                     |
|                    |                               | автором в боте   |                     |
|                    |                               | (анонимно)       |                     |
|                    |                               | Решение:         |                     |
|                    |                               | confirmed / rej. |                     |
|                    |                               |                  |                     |
|                    | Если confirmed:               |                  |                     |
|                    | -1 балл репутации получателю  |                  |                     |
|                    | (или больше — на усмотрение   |                  |                     |
|                    | модератора)                   |                  |                     |
|                    | Опционально: блокировка       |                  |                     |
|                    | (по решению Super-админа)     |                  |                     |
|                    | Уведомление получателю:       |                  | «На тебя поступила  |
|                    | без имени автора              |                  | подтверждённая      |
|                    |                               |                  | жалоба, категория Y»|
|                    | Уведомление автору:           |                  |                     |
|                    | «Жалоба рассмотрена»          |                  |                     |
|                    |                               |                  |                     |
|                    | Если rejected:                |                  |                     |
|                    | Получатель НЕ уведомляется    |                  |                     |
|                    | Автор: «Жалоба отклонена»     |                  |                     |
|                    | + опц. причина                |                  |                     |
```

### Исключения

- **E1. Жалоба на себя**: бот отвергает.
- **E2. Жалоба от деактивированного участника**: запрещена.
- **E3. Серия жалоб от одного на одного >3/мес**: бот блокирует, предлагает написать в Support.
- **E4. Жалоба содержит запрещённый контент** (угрозы, личные данные): модератор может пометить как `escalated` → Super-админ.
- **E5. Самостоятельная жалоба получателя на действия модерации**: создаётся новая жалоба, рассматривает Super-админ.

## 3. Процесс: Звёзды и репутация

### Принцип

Каждый участник имеет **скрытый числовой балл репутации** и **публичный визуальный бейдж 0–4★**. Бейдж отображается в чате (через бота с правом администрирования).

### Источники изменений балла

| Событие | Дельта балла |
|---|---|
| Получил респект | +1 |
| Подтверждённая жалоба (стандартная) | -1 (или больше по решению модератора) |
| Подтверждённая жалоба (серьёзная) | до -5, на усмотрение Super-админа |
| Месячная деградация (cron) | **открытый вопрос с командой клуба** |
| Падение от неактивности | **открытый вопрос с командой клуба** (варианты: 60 дней без сообщения / N сообщений в месяц) |
| Ручная корректировка модератором | задокументированная, с указанием причины |
| Ручная корректировка Super-админом | без ограничений по величине |

### Пороги звёзд

**Открытый вопрос с командой клуба** — конкретные пороги (`0★`, `1★`, `2★`, `3★`, `4★`) не зафиксированы.

Предлагаемая структура (на этапе Phase 2 уточнить):
```
0★ : балл < нижний_порог
1★ : нижний_порог ≤ балл < порог_2
2★ : порог_2 ≤ балл < порог_3
3★ : порог_3 ≤ балл < порог_4
4★ : балл ≥ порог_4
```

### Cron-задача: ежемесячный пересчёт

```
| Время                | Действие                                                  |
|----------------------|-----------------------------------------------------------|
| 1-го числа 03:00 МСК | Для каждого active участника:                             |
|                      |   - применить деградацию (коэффициент TBD)                |
|                      |   - проверить неактивность (механизм TBD)                 |
|                      |   - пересчитать звезду на основе порогов                  |
|                      |   - если звезда изменилась → обновить бейдж в чате клуба  |
|                      |     (через изменение прозвища/тега бота)                  |
|                      |   - уведомить участника при изменении звезды              |
|                      | AuditLog.log_task для всей задачи                         |
```

### Бизнес-правила

- Звезда отображается **только в чате клуба**, не публично за его пределами.
- Изменение звезды → push-уведомление участнику.
- При достижении 0★ возможны санкции (предупреждение / временное mute / kick — решение Super-админа).
- Звезда не влияет на возможность получать респекты, но влияет на видимость в матчинге (Phase 2 — AI-Agents Agent уточнит).

## 4. Процесс: Роли (ниши)

### CRUD ролей

- Модератор/Super-админ в web-админке: создаёт, редактирует, архивирует роли.
- Роль = {название, описание, иконка, активна/архивирована, родительская роль (опционально)}.

### Назначение роли

```
| Основатель / Модератор          | Бот                       | Участник              |
|---------------------------------|---------------------------|-----------------------|
| На интервью или после него      |                           |                       |
| → выбор основной роли           |                           |                       |
| → опц. доп. роль                |                           |                       |
| Сохранение в web-админке        |                           |                       |
|                                 | Запись в user.role        |                       |
|                                 | Уведомление участнику     | «Тебе назначена роль X»|
|                                 | Запись в AuditLog         |                       |
```

### Бизнес-правила

- Одна **основная роль** + до **2 дополнительных**.
- Смена основной роли — только Модератор/Super-админ, с обязательной причиной.
- Удаление архивной роли запрещено — только архивация (история ролей участников сохраняется).
- Роль участника доступна в его профиле в mini-app, используется в AI-матчинге (Phase 2).

## 5. Процесс: KYC-флаг (ручной)

### Триггер

Модератор/Super-админ в web-админке отмечает участника как «KYC verified» (или снимает флаг).

### Swim-lane

```
| Модератор/Super     | Web-админка       | AuditLog            | Участник                |
|---------------------|-------------------|---------------------|-------------------------|
| Получил документы   |                   |                     |                         |
| (вне системы:       |                   |                     |                         |
| Telegram/email)     |                   |                     |                         |
| Принял решение      |                   |                     |                         |
| Поставил флаг       |                   |                     |                         |
| kyc_verified=true   |                   |                     |                         |
| + комментарий       |                   |                     |                         |
|                     | save() триггерит  | log_db_update       |                         |
|                     | сигнал            | (old=false → true)  |                         |
|                     |                   |                     | (опц.) Уведомление в mini-app|
```

### Бизнес-правила

- KYC — **внеплатформенный процесс** на старте (Phase 2). Self-service KYC через провайдеров — out of scope.
- Хранение документов — **не в нашей системе**; в БД только флаг и комментарий ответственного.
- Снять флаг может только Super-админ.
- Влияние на функциональность: участники с KYC получают повышенный приоритет в матчинге (TBD AI-Agents Agent).
