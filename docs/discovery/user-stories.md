# User Stories: Клуб 33

Формат: `Как [роль], я хочу [действие], чтобы [ценность]`. Acceptance criteria — Given/When/Then. Приоритеты: **P0** (must), **P1** (should), **P2** (could). Фазы: **Ф1 / Ф2 / Ф3**.

Эпики:
1. **Воронка** (E-FUNNEL) — приём кандидатов, заявка, модерация, интервью
2. **Платежи** (E-PAY) — тарифы, оплаты, подтверждения
3. **Доступ** (E-ACCESS) — закон клуба, invite, продление, напоминания
4. **Социал** (E-SOCIAL) — роли, респекты, звёзды, жалобы, профиль
5. **Время** (E-TIME) — дар, сжигание, lifetime-бюджет, конвертер
6. **AI** (E-AI) — матчинг, RAG, digest
7. **Админка** (E-ADMIN) — управление, финансы, воронка, AI-дашборд, экспорт

## Эпик 1: Воронка (E-FUNNEL)

### US-001: Старт бота и презентация — `P0` / Ф1
**Как** кандидат, **я хочу** увидеть, что за клуб, **чтобы** решить, подавать ли заявку.
- **Given** новый Telegram-пользователь **When** отправляет `/start` **Then** бот показывает приветствие на «ты», описание клуба и кнопки «Подать заявку», «Мой статус», «Поддержка».
- **Edge:** если пользователь уже подавал заявку — кнопка «Подать заявку» сменяется на «Где моя заявка».

### US-002: Подача заявки через Google-форму — `P0` / Ф1
**Как** кандидат, **я хочу** заполнить заявку, **чтобы** меня рассмотрели.
- **Given** кандидат нажал «Подать заявку» **When** заполняет Google-форму (имя, профессия, соцсети, кто пригласил, ожидания) и отправляет **Then** backend получает webhook, создаёт `Application`, бот пишет: «Получили твою заявку, на модерации».

### US-003: Статус заявки — `P0` / Ф1
**Как** кандидат, **я хочу** проверить статус, **чтобы** не нервничать.
- **Given** заявка существует **When** кандидат жмёт «Мой статус» **Then** бот показывает текущий шаг (на модерации / приглашён на интервью / интервью назначено / одобрен / оплата ожидается / в клубе).

### US-004: Уведомление об одобрении заявки — `P0` / Ф1
**Как** кандидат, **я хочу** узнать, что заявка одобрена, **чтобы** записаться на интервью.
- **Given** админ одобрил заявку **When** меняется статус **Then** бот пишет кандидату: «Прошёл первый фильтр, выбери слот для интервью» + кнопка «Записаться».

### US-005: Запись на интервью — `P0` / Ф1
**Как** кандидат, **я хочу** выбрать удобный слот, **чтобы** встретиться с основателем.
- **Given** кандидат одобрен **When** жмёт «Записаться» **Then** открывается календарь (`club33calendar` / Calendly), выбор слота, подтверждение в боте.
- **Edge:** напоминание за день и за час; перенос/отмена через бот.

### US-006: Решение по итогам интервью — `P0` / Ф1
**Как** основатель, **я хочу** выставить решение, **чтобы** запустить оплату или закрыть заявку.
- **Given** интервью прошло **When** основатель в админке отмечает «Одобрить → к оплате» / «Отказать» / «Lifetime» **Then** статус обновляется, кандидат получает соответствующее уведомление в боте.

### US-007: Отказ кандидату — `P1` / Ф1
**Как** кандидат, **я хочу** получить понятный отказ, **чтобы** не оставаться в подвешенном состоянии.
- **Given** заявка отклонена **When** статус меняется **Then** бот пишет «не сейчас» с дружелюбной формулировкой; повторная подача — через N месяцев (см. open decision).

## Эпик 2: Платежи (E-PAY)

### US-008: Выбор тарифа — `P0` / Ф1
**Как** одобренный кандидат, **я хочу** выбрать тариф, **чтобы** оплатить вступление.
- **Given** статус «одобрен после интервью» **When** в боте «Оплатить» **Then** видит карточки: 6 мес = 1000 USD, 12 мес = 1500 USD; lifetime — недоступен к покупке.

### US-009: Оплата через USDT — `P0` / Ф1
**Как** кандидат, **я хочу** оплатить криптой, **чтобы** не зависеть от карт/банков.
- **Given** выбран USDT **When** выбирает сеть (TRC20 / TON) **Then** бот показывает адрес и сумму с памяткой; после оплаты — webhook → автоподтверждение.
- **Edge:** если >10 минут без подтверждения — кнопка «Поддержка».

### US-010: Оплата через ЮKassa (RUB) — `P0` / Ф1
**Как** кандидат, **я хочу** оплатить рублями, **чтобы** не возиться с криптой.
- **Given** выбран ЮKassa **When** генерируется инвойс **Then** курс USD→RUB фиксируется в момент создания, ссылка на оплату; webhook подтверждает.

### US-011: Оплата через СБП — `P0` / Ф1
**Как** кандидат, **я хочу** заплатить по СБП, **чтобы** использовать привычный способ.
- **Given** выбран СБП **When** оплачивает по реквизитам **Then** жмёт «Я оплатил»; админ вручную подтверждает в админке; статус обновляется, кандидат получает push.

### US-012: Кнопка «Поддержка» при зависшей оплате — `P0` / Ф1
**Как** кандидат, **я хочу** написать в поддержку, **чтобы** разрулить проблему с оплатой.
- **Given** оплата >10 минут без подтверждения **When** бот показывает «Поддержка» **Then** клик → диалог с админом + нотификация в админ-чат.

### US-013: История платежей — `P1` / Ф1
**Как** участник, **я хочу** видеть свои оплаты, **чтобы** свериться.
- **Given** есть подтверждённые платежи **When** участник в «Мой доступ → история» **Then** видит даты, суммы, метод, статус.

### US-014: Продление подписки — `P0` / Ф1
**Как** участник, **я хочу** продлить подписку, **чтобы** не потерять доступ.
- **Given** активная подписка **When** в «Мой доступ» жмёт «Продлить» **Then** выбирает тариф → оплата (как US-009/010/011) → дата окончания продлевается.

## Эпик 3: Доступ (E-ACCESS)

### US-015: Принятие закона клуба — `P0` / Ф1
**Как** оплативший кандидат, **я хочу** принять закон клуба, **чтобы** получить доступ.
- **Given** оплата подтверждена **When** бот показывает текст «закона клуба» и просит **ввести фразу вручную** **Then** при точном совпадении (без учёта регистра/пробелов по краям) — переход к выдаче invite. **Кнопка «Согласен» отсутствует.**
- **Edge:** при неверном вводе — мягкая подсказка «попробуй ещё раз»; счётчик попыток, после 5 — уведомление админу.

### US-016: Получение invite-ссылок — `P0` / Ф1
**Как** новый участник, **я хочу** получить ссылки в канал и чат, **чтобы** войти.
- **Given** закон принят **When** бот генерирует две одноразовые ссылки (лимит 1 использование, TTL 24 ч) **Then** присылает их кандидату; запись в `sessions`.

### US-017: Перевыпуск истёкшего invite — `P1` / Ф1
**Как** новый участник, **я хочу** запросить новую ссылку, **чтобы** войти, если первая истекла.
- **Given** ссылка использована/истекла, но участник не вошёл **When** жмёт «Получить ссылку заново» **Then** новая одноразовая ссылка с тем же TTL.

### US-018: Раздел «Мой доступ» — `P0` / Ф1
**Как** участник, **я хочу** видеть статус и срок, **чтобы** не пропустить продление.
- **Given** участник авторизован **When** открывает «Мой доступ» **Then** видит тариф, дату окончания, кнопки «Продлить», «Поддержка».
- **Lifetime:** плашка «lifetime», без «Продлить».

### US-019: Напоминание о продлении — `P0` / Ф1
**Как** участник, **я хочу** получать напоминания за 10 дней, **чтобы** успеть продлить.
- **Given** до окончания подписки ≤ 10 дней **When** scheduler пробегает по подпискам **Then** бот шлёт «До конца подписки X дней» + кнопка «Продлить». Повтор за 3 дня и в день окончания.
- **Lifetime:** напоминания **не отправляются**.

### US-020: Истёкшая подписка — `P0` / Ф1
**Как** участник с истёкшей подпиской, **я хочу** понимать, что случилось, **чтобы** восстановиться.
- **Given** дата окончания прошла **When** scheduler срабатывает **Then** доступ закрывается (исключение из канала/чата по политике клуба), бот пишет «Подписка закончилась, можешь продлить».

## Эпик 4: Социал (E-SOCIAL)

### US-021: Профиль участника в mini-app — `P0` / Ф2
**Как** участник, **я хочу** видеть и редактировать профиль, **чтобы** меня находили.
- **Given** mini-app открыт **When** «Профиль» **Then** видит имя, роль, звёзды, bio, компетенции; bio и компетенции можно редактировать.

### US-022: Создание роли (ниши) — `P0` / Ф2
**Как** админ, **я хочу** создать роль, **чтобы** назначать её на интервью.
- **Given** в админке раздел «Роли» **When** «+ Создать», ввод имени, флаг `kyc_required` **Then** роль появляется в списке.

### US-023: Назначение роли на интервью — `P0` / Ф2
**Как** основатель, **я хочу** назначить кандидату роль, **чтобы** он встроился в клуб.
- **Given** интервью прошло **When** в карточке кандидата выбирается роль из списка **Then** роль сохраняется, mini-app отражает её в профиле.

### US-024: Респект через mini-app — `P0` / Ф2
**Как** участник, **я хочу** дать респект другому, **чтобы** поблагодарить.
- **Given** месячный баланс ≥ 1 и получателю в этом месяце ещё дано < 3 **When** в карточке участника жмёт «Респект» **Then** счётчик уменьшается, у получателя — +1.
- **Edge 1:** баланс 0 → кнопка disabled, тост «Респекты вернутся 1 числа».
- **Edge 2:** получателю уже дано 3 в этом месяце → кнопка disabled, тост «Уже 3 в этом месяце».

### US-025: Респект через команду /respect — `P1` / Ф2
**Как** участник, **я хочу** дать респект через бот, **чтобы** не открывать mini-app.
- **Given** в чате клуба **When** делает reply на сообщение и пишет `/respect [причина]` **Then** бот выдаёт респект автору reply с теми же лимитами.

### US-026: Сброс баланса респектов — `P0` / Ф2
**Как** система, **я хочу** обнулять балансы 1 числа в 00:00 МСК, **чтобы** месяц начинался с чистого листа.
- **Given** 1-е число 00:00 МСК **When** scheduler срабатывает **Then** все балансы → 30, лимиты «3 на пользователя» обнуляются.

### US-027: Анонимность респекта для получателя — `P0` / Ф2
**Как** получатель респекта, **я хочу** видеть, что мне дали респект, но не от кого, **чтобы** не возникало неловкости.
- **Given** получен респект **When** открываю профиль **Then** вижу счётчик/историю **без имени дарящего**.
- **Админ:** видит полную историю.

### US-028: Жалоба на участника — `P1` / Ф2
**Как** участник, **я хочу** пожаловаться, **чтобы** модератор отреагировал.
- **Given** в mini-app открыта карточка участника **When** жмёт «Пожаловаться», заполняет причину **Then** жалоба создаётся, для получателя **анонимна**, модератор получает её в админке.

### US-029: Рассмотрение жалобы — `P0` / Ф2
**Как** модератор, **я хочу** обработать жалобу, **чтобы** держать тон чата.
- **Given** в админке очередь жалоб **When** открывает карточку **Then** видит причину, историю обоих участников, кнопки «подтвердить / отклонить / эскалировать админу»; решение логируется.

### US-030: Звёзды и репутация — `P0` / Ф2
**Как** участник, **я хочу** видеть свои звёзды, **чтобы** понимать вес в клубе.
- **Given** есть респекты/вклад **When** открываю профиль **Then** вижу звёзды (0–4) по моей роли + score; формула задана в админке (Open Decision).

### US-031: Падение от неактивности — `P1` / Ф2
**Как** клуб, **я хочу** снижать звёзды у неактивных, **чтобы** репутация была живой.
- **Given** прошло 60 дней без активности (или N сообщений — см. Open Decision) **When** scheduler пересчитывает **Then** баллы понижаются согласно правилу, звёзды могут упасть.

## Эпик 5: Время (E-TIME)

### US-032: Дар времени — `P0` / Ф3
**Как** участник с активной подпиской, **я хочу** подарить дни другому, **чтобы** поддержать.
- **Given** в моей подписке ≥ 30 дней до окончания **When** в mini-app «Подарить дни» → выбор получателя и N дней **Then** N дней списываются у меня, добавляются получателю; запись в `time_gifts`.
- **Edge:** если остаток после дара станет < 30 — кнопка disabled с пояснением.
- **Команда `/gift @user N`** — то же поведение.

### US-033: Сжигание дней — `P1` / Ф3
**Как** участник, **я хочу** сжечь дни в пользу клуба, **чтобы** выразить признательность.
- **Given** есть подписка **When** `/burn N` или кнопка в mini-app **Then** подтверждение → N дней списываются, запись в `day_burns`.

### US-034: Lifetime годовой бюджет 33 дня — `P0` / Ф3
**Как** lifetime-участник, **я хочу** видеть свой годовой бюджет 33 дня для дарения, **чтобы** распоряжаться им.
- **Given** статус lifetime **When** открываю «Мой доступ» **Then** вижу плашку «годовой бюджет: использовано X / 33».
- **Edge:** 1 января 00:00 МСК scheduler обнуляет счётчик и восстанавливает 33 дня.

### US-035: Админ-конвертер «доллары ↔ дни» — `P0` / Ф3
**Как** админ, **я хочу** вручную начислить или списать дни пользователю, **чтобы** разрулить нестандартные случаи.
- **Given** в админке карточка пользователя **When** «Начислить/списать дни», N + комментарий + автоматический USD-эквивалент (1 день = 2.5 USD) **Then** запись в `admin_day_adjustments`, изменение даты окончания, лог.

### US-036: Реакция-стикер как респект — `P1` / Ф3
**Как** участник, **я хочу** дать респект реакцией-стикером, **чтобы** не открывать mini-app.
- **Given** в чате клуба **When** ставлю особый стикер-реакцию на сообщение **Then** автору +1 респект, мой баланс −1, лимит «3 на получателя» работает.

## Эпик 6: AI (E-AI)

### US-037: AI-матчинг /match — `P0` / Ф2
**Как** участник, **я хочу** найти полезного человека по запросу, **чтобы** решить задачу.
- **Given** заполнено bio + роль **When** делаю `/match <запрос>` (или кнопка в mini-app) **Then** Claude возвращает топ-3 участника из той же или смежной ниши с обоснованием. Учитываются роль, bio, компетенции, embedding 1536.
- **Edge:** если профиль пуст — подсказка «заполни bio, чтобы тебя находили».

### US-038: Оценка качества матча — `P1` / Ф2
**Как** клуб, **я хочу** собирать CSAT после /match, **чтобы** улучшать модель.
- **Given** /match выдал результат **When** через час или после контакта **Then** mini-app предлагает оценить 1–5 + комментарий.

### US-039: База знаний /ask — `P0` / Ф2
**Как** участник, **я хочу** спросить о клубе/чатах, **чтобы** не листать историю.
- **Given** Listener индексирует чат **When** делаю `/ask <вопрос>` **Then** Claude отвечает на основе KB + указывает источники (ссылки на сообщения).
- **Edge:** если ответа нет — «не нашёл, попробуй переформулировать».

### US-040: Индексация чата Listener-ботом — `P0` / Ф2
**Как** система, **я хочу** индексировать чат клуба, **чтобы** /ask и digest работали.
- **Given** Listener-бот в чате, согласие участников задано в «законе клуба» **When** появляется новое сообщение **Then** оно сохраняется в `chat_messages`, чанки → `kb_chunks` (embedding 1536).

### US-041: Еженедельный digest — `P1` / Ф2
**Как** участник, **я хочу** получать раз в неделю выжимку чата, **чтобы** быть в курсе.
- **Given** понедельник утром **When** scheduler срабатывает **Then** Claude генерирует summary за неделю (темы, открытые запросы, решения), сохраняет в `daily_summaries`, рассылает по подписке + доступно по `/digest`.

### US-042: AI-дашборд (минимум) — `P1` / Ф2
**Как** админ, **я хочу** видеть расходы и метрики AI, **чтобы** контролировать бюджет.
- **Given** в админке раздел AI **When** открываю **Then** вижу запросы по фичам, токены, $ inference, графики по неделям.

## Эпик 7: Админка (E-ADMIN)

### US-043: Список пользователей — `P0` / Ф1
**Как** админ, **я хочу** искать и фильтровать пользователей, **чтобы** оперативно отвечать.
- **Given** есть пользователи **When** ввожу `username` или `telegram_id` **Then** список с фильтрами (роль, статус, тариф, KYC, период оплат) + пагинация.

### US-044: Карточка пользователя — `P0` / Ф1
**Как** админ, **я хочу** видеть всё о пользователе, **чтобы** принимать решения.
- **Given** карточка пользователя **When** открываю **Then** вижу: статус, тариф, оплаты, LTV, роль, звёзды, действия (lifetime, продление, разблокировка, новый invite).

### US-045: Очередь оплат — `P0` / Ф1
**Как** админ, **я хочу** видеть статусы оплат, **чтобы** подтверждать СБП и разбирать проблемы.
- **Given** есть платежи **When** «Оплаты» **Then** фильтры (ожидание / подтверждено / отклонено / возврат / просрочено); по СБП — кнопка «Подтвердить вручную» с комментарием.

### US-046: Дашборд воронки — `P0` / Ф1
**Как** админ, **я хочу** видеть конверсии, **чтобы** улучшать воронку.
- **Given** есть события **When** открываю «Воронка» **Then** вижу `bot_start → application → approved → payment_completed → invite_used` с долями и drop-off; фильтры по источнику и периоду.
